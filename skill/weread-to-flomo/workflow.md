# 导出与浏览工作流

本文档覆盖三类操作：浏览（read-only）、单本/全部导出、单条导出。任何调用前先按 `SKILL.md` 的「启动流程」做完两侧依赖探测，**遇到第一个失败立即停止**。

---

## 0. 通用前置（每次调用开头都做）

**强制顺序，不要跳步、不要并发，不要因为「上下文里看起来工具都在」就省略**：

1. **完整执行 `SKILL.md` 启动流程的全部 6 状态探测**（先 weread 后 flomo）。**两侧都通过**之前不要碰任何 weread API、也不要碰任何 flomo MCP 工具（除了用作探活的 `get_format_guide`）。
   - weread 探测：尝试一次最小 `/user/notebooks count=1` 请求；按返回码/网络错误分支映射到 SKILL.md 的 3 个 weread 状态文案
   - flomo 探测：调一次 `get_format_guide`（无副作用），按返回成功/鉴权失败/工具不可见映射到 3 个 flomo 状态文案
   - 一个常见反模式：「我看到 `mcp__flomo__memo_create` 在工具列表里就当 flomo 已经 ready 了，不再调 `get_format_guide`」—— **不行**。工具在列表里只代表 transport 已连接，不代表 token 仍然有效或服务端实际可用。`get_format_guide` 是廉价探活，必须做。
2. 读取 weread-skills 的当前 `version`（在其 `SKILL.md` frontmatter 里），后续每次调 weread API 都把它当 `skill_version` 传入；**不要硬编码版本号**。
3. **检查上次的失败回执**（探活通过后、用户意图分流前必做）：
   ```
   memo_search keywords:"wr:fail" tag:"微信读书/导出失败" limit:10
   ```
   如果命中（去 escape 后的 `content` 含 `[wr:fail:...]`），把每条回执里列出的 `bookmarkId`/`reviewId` 提取出来，**主动告诉用户**：
   > 上次导出有 N 条未完成（来自《XX》导出会话）。要不要先重试这些？
   > - 回复「重试」→ 用收集到的 ID 集合，按 §3 单条流程逐条重试，**重试成功后，把对应的失败回执 memo 标记为已处理（追加一行 `[resolved]` 到内容里，但本版本只追加文字、不改去重标记）**
   > - 回复「跳过」→ 直接走本次用户的新意图
4. 解析用户意图（见下面「意图路由」），不要静默选默认范围。

> 这一节的强约束是为了避免一个真实发生过的失败：浏览模式下，agent 看到「weread 通了」就直接拉了书架数据展示给用户；选完书后才发现 flomo MCP 根本没配，用户做了无用的浏览操作还被打断。**只要本次会话可能涉及 flomo 写入，就必须在第一步把两侧都探完。**

## 1. 意图路由

| 用户表达 | 路由到 |
|---------|--------|
| 「列一下我有笔记的书」「先看看再决定」「我都有哪些划线」 | §2 浏览 |
| 「把这条划线/想法导出到 flomo」「上面那条搬到浮墨」 | §3 单条导出 |
| 「把《XX》第 N 章导出到 flomo」「这一章的笔记搬到浮墨」「chapter export」 | §3.5 章节导出 |
| 「把《XX》的笔记导出到 flomo」「同步《YY》的划线和想法」 | §4 单本导出 |
| 「全量导出」「所有书的笔记都同步到 flomo」「书架里有笔记的都导」 | §5 全部导出 |
| 仅说「导出笔记到 flomo」「同步到 flomo」（范围不明） | §1.1 模糊路由 |

### 1.1 模糊路由（必做）

如果用户没有指明范围，**不要**自行选默认。先调 `/user/notebooks`（`count=20`）拿一页概览，按 §2.1 表格展示，然后给出 4 选 1 提示：

> 你想：
> 1. 导出**某本书**的全部笔记 — 回复书名或序号
> 2. 导出**某一章**的笔记 — 回复 "《书名》第 N 章" 或 "《书名》章节标题"
> 3. 导出**某一条**特定划线/想法 — 回复 "划线: 内容片段" 或 "想法: 内容片段"
> 4. **全部导出**（所有 N 本有笔记的书） — 回复 "全部"

按用户回答路由到 §3/§3.5/§4/§5。

## 2. 浏览模式（read-only）

**强约束**：浏览过程中**绝不**调用 `memo_create` 或 `memo_search`。只有用户在 §2.3 明确选择「导出」后才进入对应导出流程。

### 2.1 列书（List）

调 weread `/user/notebooks` 翻页：

```json
{"api_name":"/user/notebooks","count":100,"skill_version":"<weread skill_version>"}
```

下一页带 `lastSort`（取上一页 `books[]` 最后一项的 `sort` 值）。**禁止**用 `offset`/`limit`/`params` 嵌套，会触发 weread-skills 的分页参数误用。

如果用户只说「前 N 本」就只取一页（`count=N`，最大 100），不要循环。

#### 概览展示格式

按每本书 `reviewCount + noteCount + bookmarkCount` 降序排序，展示前 ~20 本：

```
📊 你的微信读书笔记概况
- 有笔记的书籍：N 本（更多页未拉时也要明示）
- 笔记总数：M 条（划线 a + 想法 b + 书签 c）

| # | 书名 | 划线 | 想法 | 书签 | 总数 | 进度 |
|---|------|------|------|------|------|------|
| 1 | 《XX》 - 作者 | 12 | 3 | 0 | 15 | 100% |
| 2 | 《YY》 - 作者 | 5 | 0 | 0 | 5 | 60% |
| ... | ... | ... | ... | ... | ... | ... |
```

字段口径：
- 划线 = `noteCount`，想法 = `reviewCount`，书签 = `bookmarkCount`，总数 = 三者相加
- 进度 = `readingProgress`，0–100 的整数，**展示时务必带 % 号**（参考 weread-skills 的 `book.md`）
- 书名前用「《》」包裹

最后追加：

> 想查看哪本书的划线和想法？回复书名或序号即可。也可以说「导出全部」或「先不导」。

### 2.2 下钻到具体一本（Drill）

用户按书名或序号选定后，并行调两个接口：

```json
{"api_name":"/book/bookmarklist","bookId":"<bookId>","skill_version":"<v>"}
```

```json
{"api_name":"/review/list/mine","bookid":"<bookId>","count":100,"skill_version":"<v>"}
```

> ⚠️ 接口参数名不一致：`/book/bookmarklist` 用大写 `bookId`，`/review/list/mine` 用全小写 `bookid`。**别写错**。

`/review/list/mine` 翻页：`hasMore=1` 时下一页带 `synckey`，循环到 `hasMore=0` 拼齐所有 `reviews[]`。

把两边数据按 `chapterUid` 用 `chapters[]`（来自 `/book/bookmarklist` 的同次返回）查 `title`，按章节分组。

#### ⚠️ Chapter identifier 三胞胎区分

WeRead 在不同字段里给了**三种章节标识**，构造数据时极易写错。**绝对要分清**：

| 字段 | 例子 | 用途 |
|------|------|------|
| `chapter.chapterUid` | `42` | 内部 ID，**用于 `weread://` URL 的 `chapterUid` 参数、用于跨接口关联**（`updated[].chapterUid` / `review.chapterUid` 也是这个 ID） |
| `chapter.chapterIdx` | `40` | 章节序号（人读的「第 40 章」中的 40），通常 ≠ `chapterUid` |
| `chapter.title` | `第40章` | 人类可读章节名，**用于展示** |
| `bookmarkId` 中段 | `674073_40_2826-2841` 中的 `40` | 是章次/键，**不是 `chapterUid`**，**禁止解析使用** |

规则：
- URL 拼接 → 用 `chapterUid`
- 标题文本展示 → 用 `title`
- `bookmarkId` → 只用于去重原样回传，绝不要切片解析其内部数字
- 跨数据关联 → 用 `chapterUid`（如 review 与 bookmark 在同一章 = `chapterUid` 相等）

#### 下钻展示格式（聊天界面渲染，不是写入 flomo 的内容）

```
📖 《<书名>》 — <作者>
📊 X 条划线，Y 条想法

### 第 N 章「<章节标题>」（n 条）

> <划线原文>

> <另一条划线原文>
> 想法：<针对这条划线的想法正文>

### 第 N+1 章「<下一章节标题>」（k 条）

> ...
```

注意：
- 这里使用 markdown `>` blockquote 是**给聊天 UI 的**，不是写入 flomo 的内容。flomo 不支持 `>`。
- `markText` 与 `review.content` 都做去首尾空白后展示，内部换行原样保留。
- `chapterUid` 在 `chapters[]` 找不到对应项时，章节标题写「未知章节」。
- 整本书评（`isFinish=1` 且没有 `chapterUid`）放在最后一个「全书」段。
- 长内容**不要**截断。逐字呈现。

### 2.3 下钻后的菜单

展示完笔记后，**必须**提示：

> 要把这些导出到 flomo 吗？
> - 回复「全部导出」→ 整本书的划线 + 想法都写入 flomo（流程见 §4）
> - 回复「只导划线」/「只导想法」→ 按类型筛选导出
> - 回复具体一条的引用片段 → 单条导出（流程见 §3）
> - 回复「换一本」→ 回到列表
> - 回复「先不导」→ 结束

「换一本」时，把刚才已拉的 `chapters[]`/`bookmarklist`/`reviews[]` 数据丢弃，重新提示用户从概览选；不要默认地循环。

## 3. 单条导出

适用：用户在浏览或别处见到某条具体划线/想法，明确要导这一条。

### 3.1 定位

如果上下文里已经能找到具体的 `bookmarkId`（划线）或 `reviewId`（想法），直接用。

否则让用户贴片段：

> 请贴一下你想导出的内容片段（10 个字以内即可），或者告诉我书名 + 章节，我帮你定位。

收到片段后：
1. 如果用户说了书名 → 先按 §4.1 拿到 `bookId`
2. 调 `/book/bookmarklist`（划线）或 `/review/list/mine`（想法），在返回的 `markText` / `review.content` 里做子串匹配
3. 多个候选时编号让用户选；零个时说「没找到，请确认书名或贴更具体的片段」

#### 3.1.1 候选列表渲染规则（硬约束）

向用户出示候选列表（无论来自子串匹配、章节内全部条目，还是单条选择菜单）时，**必须一次性渲染完整片段**：

- 每条候选的预览文本控制在 **40 字以内**（中文按字符算），超长用 `…` 截断尾部；不要截首
- **禁止**先发占位行（如「待显示」「TBD」「内容稍后」）再补全。这会触发用户「这是什么意思？」的反复澄清，把单次导出变成多回合 chat
- **禁止**只渲染前 1–3 条然后省略后面的「以此类推」；列表不长（≤15 条）就全展示
- 若条目 >15，按章节/类型先分桶，每桶展示前 5 条 + 「该章节还有 N 条，要全展示吗？」提示
- 渲染来源同 `format.md` §8.1：从临时文件 / 脚本返回值取字面字节，不要凭印象拼装片段

理由：候选列表是用户做选择的依据。占位符相当于让用户在信息不全的状态下做决策，必然触发追问；agent 的「先给框架再补内容」协作模式适用于规划文档，不适用于选择菜单。

### 3.2 去重

按 `dedup.md` 的方式调一次 `memo_search`，命中即告诉用户「这条已经在 flomo 里了，跳过」并停止。

### 3.3 写入

按 `format.md` 渲染单条 memo 模板，调 `memo_create`，把生成的 memo 信息汇报给用户。

## 3.5 章节级导出

适用：用户给出**书 + 章节**（"《XX》第 40 章"、"《YY》序章"、"chapter 7"），只导那一章的划线 + 想法。

### 3.5.1 识别书 + 章节

1. **识别书**：按 §4.1 拿 `bookId`（用户给书名 → `/store/search`；用户给 bookId → 直接用）
2. **拉一次 `/book/bookmarklist`**：拿到 `chapters[]`（如果上下文里已经有该 bookId 的 `chapters[]`，按 `SKILL.md` 的「上下文复用」规则直接用，不要重复调）
3. **匹配章节**（在 `chapters[]` 上做）：
   - 用户说「第 N 章」/「Chapter N」/「第 N 集」 → 在 `chapter.title` 里子串匹配（如 "第40章"）；如果有 `chapterIdx`，也可作为辅助匹配字段
   - 用户说章节标题片段（"序章"、"黑暗森林"） → 同样在 `title` 里子串匹配
   - 找到 **1 个** → 取它的 **`chapterUid`**（**不是** `chapterIdx`、**不是** `title` 里的数字）
   - 找到 **0 个** → 列出该书全部 `chapters[]`（编号 + 标题），让用户挑序号
   - 找到 **多个** → 编号让用户选

### 3.5.2 过滤 + 去重

按选中的 `chapterUid` 过滤本地数据（不需要再调接口）：

```js
const targetUid = selectedChapter.chapterUid;
const chapterBookmarks = bookmarklist.updated.filter(b => b.chapterUid === targetUid);

// 想法需要并行调 /review/list/mine 拿全（翻页直到 hasMore=0），再过滤
const chapterReviews = allReviews.filter(r => r.review.chapterUid === targetUid);
// 整本书评（无 chapterUid 或 isFinish=1 且无 chapterUid）不属于任何章，章节导出**不包含**整本书评
```

去重：**仍然按全书范围**调一次 `dedup.md` 的批量探测（`tag:"微信读书/{清洗后书名}"`），本地比对当前章节的子集。理由：避免章节内某条划线已经被「单本导出」写过却看不到。

### 3.5.3 清单 + 写入

清单展示：

```
准备导出《<书名>》· <章节标题>：
- 划线 X 条 → 新增 a 条，跳过 b 条（已存在）
- 想法 Y 条 → 新增 c 条，跳过 d 条
- 标签：#微信读书 #微信读书/划线 或 #微信读书/想法 #微信读书/{清洗后书名}
```

后续走 §4.3 的 fast-path 判定：
- 待新增 ≤ 5 → 跳过用户确认，按 §4.3 fast-path 展示 1 条样本 memo 后直接写入
- 待新增 > 5 → 等待用户「确认」回复

写入流程同 §4.4，章节标题用 `chapter.title` 渲染入 memo 正文。

## 4. 单本导出

### 4.1 识别书

- 用户给 bookId → 直接用
- 用户给书名 → 调 `/store/search`：
  ```json
  {"api_name":"/store/search","keyword":"<书名>","scope":10,"skill_version":"<v>"}
  ```
  从 `results[0].books` 取前 3 个候选，编号给用户选；只有 1 个就直接用并告知。

### 4.2 拉笔记

并行调用：
- `/book/bookmarklist` — 取划线 `updated[]` 与 `chapters[]`
- `/review/list/mine` — 取想法 `reviews[]`，翻页直到 `hasMore=0`
- 书名：取 `/book/bookmarklist` 返回的 `book.title`；若缺，再调 `/book/info` 补一次

### 4.3 去重 + 清单

按 `dedup.md` 的批量优化方式：先 `memo_search keywords:"wr:hl" tag:"微信读书/{清洗后书名}" limit:200` 拿到这本书已存在 memo 的 ID 集合，本地比对每条划线/想法。

⚠️ **批量 probe 索引延迟**：`memo_search` 标签过滤路径对刚写入的 memo 有几十分钟级别的不可见期（实证）。批量 probe 拿到的 `existing` 集合**必须**叠加一层 in-session 内存集合（每次 `memo_create` 成功后立即把 `(类型, ID)` 加入），否则同一会话连写两次会重复。详见 `dedup.md` §4.4。

把汇总展示给用户：

```
准备导出《<书名>》：
- 划线 X 条 → 新增 a 条，跳过 b 条（已存在）
- 想法 Y 条 → 新增 c 条，跳过 d 条
- 标签：#微信读书 #微信读书/划线 或 #微信读书/想法 #微信读书/{清洗后书名}

是否继续？（回复「确认」开始写入）
```

收到「确认 / 导出 / OK」之前**绝不**写入。

#### 小批量 fast-path

当**待新增条数 ≤ 5** 时，跳过用户确认步骤，但仍在写入前展示 **1 条样本 memo**（按 `format.md` 模板完整渲染好的全文），让用户能直接看到效果。用户有任何异议（"等等"、"不对"、"格式不对"、"先停一下"）即停止。

样本展示模板：

```
准备写入 N 条 memo（待新增 ≤ 5，跳过确认环节）。第一条样本如下：

{完整渲染好的样本 memo 正文，包括三个标签和 [wr:hl:...] 标记}

直接开始写入。
```

理由：
- 4 条以内的导出（如「这一章 4 条划线」），让用户对着数字点「确认」是无意义摩擦
- 但用户最关心的是「格式对不对」，所以**必须**展示一条真实渲染样本，而不仅仅是数字
- 留下"先停一下"逃生口让用户能在样本看出问题时立即叫停

阈值 ≤ 5 是经验值；可根据用户反馈在未来版本调整。

### 4.4 写入

按 `format.md` 模板逐条 `memo_create`：

```
tool: memo_create
arguments:
  content: <按模板渲染的正文，必须来自一次性渲染产物，不能从对话预览回打>
```

⚠️ **content 不能回打**：渲染好的 memo 字符串必须先落到临时文件（如 `/tmp/<book>_memos.json`），写入时通过 `Read` 工具取出字面字节再喂给 `memo_create`。绝对不要从 agent 自己刚才在对话或工具结果里打印的预览中复制粘贴重输——中文弯引号 `""` / 长破折号 `—` / 中文省略号 `…` 极易被静默替换成 ASCII 等价物，违反 SKILL.md 核心规则 5「不擅自变形」。详见 `format.md` §8.1。

⚠️ **写完立即 audit**：每条 `memo_create` 返回后，立刻 `memo_batch_get` 取回 + 跑 byte-exact 对比脚本，发现偏差立即按 SKILL.md「写错时的修正流程」处理。详见 `format.md` §8.1.2。批量场景的抽样规则也在 §8.1.2。

不传 `created_at`；让 flomo 用当前时间，方便用户在「每日回顾」里看到本次导出。

错误处理：
- 单条失败 → 等 2 秒指数退避重试一次。仍失败计入 `failed[]`，继续下一条
- 章节标题缺失 → 写「未知章节」占位，不跳过
- 连续 5 次 `memo_create` 全失败 → 停止整批，告知用户，**并按 `format.md` §11 写入一条「失败回执」memo**（这条回执失败本身**不计入** `failed[]`，也不再触发 5 次终止）

### 4.5 汇报

```
✓ 《<书名>》导出完成
- 新增：N 条（划线 a / 想法 b）
- 跳过：M 条（已存在）
- 失败：K 条
查看：https://flomoapp.com/mine
```

`failed[]` 非空时列出 `bookmarkId/reviewId` + 错误摘要。

#### 回显内容时去 escape

向用户回显 memo 内容（如样本预览、失败明细中的 memo 片段、跳过条目的内容预览）时，**必须**先把 flomo MCP 回包里的反斜杠 escape 还原回干净形式：

```js
const display = memo.content
  .replace(/\\\[/g, '[')
  .replace(/\\\]/g, ']')
  .replace(/\\_/g, '_');
```

否则用户在对话里看到的是 `\[wr:hl:674073\_40\_2826-2841\]` 这种脏字符串，会误以为写入有问题。flomo App 渲染层会自动脱去这些反斜杠，所以**写入本身没问题**——只是 MCP 回包里多了一层 escape。**写入新 memo 时不要做反向 escape**（写入流不需要 `\[`），这条规则只针对**回显**。

## 5. 全部导出

### 5.1 拉全部笔记本

按 §2.1 调 `/user/notebooks` 翻页，**循环到 `hasMore=0`**（不要只取一页）。

### 5.2 总清单

对每本书都按 `dedup.md` 的批量优化跑一次 `memo_search`（标签为 `微信读书/{清洗后书名}`），构造每本书的「待新增」数。

汇总展示：

```
准备全量导出（共 N 本有笔记的书）：

| # | 书名 | 待新增 | 已存在跳过 |
|---|------|--------|-----------|
| 1 | 《XX》 | 38 | 4 |
| 2 | 《YY》 | 12 | 0 |
| ... | ... | ... | ... |

合计：待新增 a 条 / 已存在 b 条 / 共预计调用 memo_create a 次。

是否继续？（回复「确认」开始；也可以回「跳过 #2 #5」「只导前 10 本」之类微调）
```

**必须**用户显式确认；从不静默跑全量。

### 5.3 逐本写入

按用户确认后的清单**逐本**进入 §4.4 的写入流程。每完成一本立刻按 §4.5 格式汇报一次（不要等全部跑完），让用户能中途打断。

每本之间不需要 sleep；如果连续 5 次 `memo_create` 全失败，停止整批，**按 `format.md` §11 写入一条失败回执 memo**，告诉用户。失败回执 memo 的写入失败本身不计入 5 次终止，也不再重试。

### 5.4 全量汇报

全部跑完后，给最终汇总：

```
✓ 全量导出完成
- 涉及书籍：N 本
- 新增：a 条（划线 x / 想法 y）
- 跳过：b 条
- 失败：c 条
```

## 6. 通用规约

- weread 接口的所有参数**平铺**在 JSON body 顶层，与 `api_name`、`skill_version` 同级；不要包在 `params`/`data`/`body` 对象里
- 时间戳字段（`createTime`、`updateTime`、`finishTime`）展示时转 `YYYY-MM-DD`，不要直接显示原始数字
- 阅读进度 `readingProgress` 是 0–100 整数，展示时带 `%`
- 所有跨次会话的内容（如用户已选书名、bookId、概览）只保留在当前对话上下文中；不要写到 flomo memo 之外的任何持久化位置
