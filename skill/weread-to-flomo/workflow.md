# 导出与浏览工作流

本文档覆盖三类操作：浏览（read-only）、单本/全部导出、单条导出。任何调用前先按 `SKILL.md` 的「启动流程」做完两侧依赖探测，**遇到第一个失败立即停止**。

---

## 0. 通用前置（每次调用开头都做）

**强制顺序，不要跳步、不要并发，不要因为「上下文里看起来工具都在」就省略**：

1. **完整执行 `SKILL.md` 启动流程的全部 6 状态探测**（先 weread 后 flomo）。**两侧都通过**之前不要碰任何 weread API、也不要碰任何 flomo MCP 工具（除了用作探活的 `get_format_guide`）。
   - weread 探测：尝试一次最小 `/user/notebooks count=1` 请求；按返回码/网络错误分支映射到 SKILL.md 的 3 个 weread 状态文案
   - flomo 探测：调一次 `get_format_guide`（无副作用），按返回成功/鉴权失败/工具不可见映射到 3 个 flomo 状态文案
   - 一个常见反模式：「我看到 `mcp__flomo__memo_create` 在工具列表里就当 flomo 已经 ready 了，不再调 `get_format_guide`」—— **不行**。工具在列表里只代表 transport 已连接，不代表 token 仍然有效或服务端实际可用。`get_format_guide` 是廉价探活，必须做。
2. 读取 weread-skills 的当前 `version`（在其 `SKILL.md` frontmatter 里），后续每次调 weread API 都把它当 `skill_version` 传入；**不要硬编码版本号**。
3. 解析用户意图（见下面「意图路由」），不要静默选默认范围。

> 这一节的强约束是为了避免一个真实发生过的失败：浏览模式下，agent 看到「weread 通了」就直接拉了书架数据展示给用户；选完书后才发现 flomo MCP 根本没配，用户做了无用的浏览操作还被打断。**只要本次会话可能涉及 flomo 写入，就必须在第一步把两侧都探完。**

## 1. 意图路由

| 用户表达 | 路由到 |
|---------|--------|
| 「列一下我有笔记的书」「先看看再决定」「我都有哪些划线」 | §2 浏览 |
| 「把这条划线/想法导出到 flomo」「上面那条搬到浮墨」 | §3 单条导出 |
| 「把《XX》的笔记导出到 flomo」「同步《YY》的划线和想法」 | §4 单本导出 |
| 「全量导出」「所有书的笔记都同步到 flomo」「书架里有笔记的都导」 | §5 全部导出 |
| 仅说「导出笔记到 flomo」「同步到 flomo」（范围不明） | §1.1 模糊路由 |

### 1.1 模糊路由（必做）

如果用户没有指明范围，**不要**自行选默认。先调 `/user/notebooks`（`count=20`）拿一页概览，按 §2.1 表格展示，然后给出 3 选 1 提示：

> 你想：
> 1. 导出**某本书**的全部笔记 — 回复书名或序号
> 2. 导出**某一条**特定划线/想法 — 回复 "划线: 内容片段" 或 "想法: 内容片段"
> 3. **全部导出**（所有 N 本有笔记的书） — 回复 "全部"

按用户回答路由到 §3/§4/§5。

## 2. 浏览模式（read-only）

**强约束**：浏览过程中**绝不**调用 `memo_create` 或 `memo_search`。只有用户在 §2.3 明确选择「导出」后才进入对应导出流程。

### 2.1 列书（List）

调 weread `/user/notebooks` 翻页：

```json
{"api_name":"/user/notebooks","count":100,"skill_version":"<weread skill_version>"}
```

下一页带 `lastSort`（取上一页 `books[]` 最后一项的 `sort` 值）。**禁止**用 `offset`/`limit`/`params` 嵌套，会触发 weread-skills 的分页参数误用。

如果用户只说「前 N 本」就只取一页（`count=N`，最大 100），不要循环。

#### 概览展示格式

按每本书 `reviewCount + noteCount + bookmarkCount` 降序排序，展示前 ~20 本：

```
📊 你的微信读书笔记概况
- 有笔记的书籍：N 本（更多页未拉时也要明示）
- 笔记总数：M 条（划线 a + 想法 b + 书签 c）

| # | 书名 | 划线 | 想法 | 书签 | 总数 | 进度 |
|---|------|------|------|------|------|------|
| 1 | 《XX》 - 作者 | 12 | 3 | 0 | 15 | 100% |
| 2 | 《YY》 - 作者 | 5 | 0 | 0 | 5 | 60% |
| ... | ... | ... | ... | ... | ... | ... |
```

字段口径：
- 划线 = `noteCount`，想法 = `reviewCount`，书签 = `bookmarkCount`，总数 = 三者相加
- 进度 = `readingProgress`，0–100 的整数，**展示时务必带 % 号**（参考 weread-skills 的 `book.md`）
- 书名前用「《》」包裹

最后追加：

> 想查看哪本书的划线和想法？回复书名或序号即可。也可以说「导出全部」或「先不导」。

### 2.2 下钻到具体一本（Drill）

用户按书名或序号选定后，并行调两个接口：

```json
{"api_name":"/book/bookmarklist","bookId":"<bookId>","skill_version":"<v>"}
```

```json
{"api_name":"/review/list/mine","bookid":"<bookId>","count":100,"skill_version":"<v>"}
```

> ⚠️ 接口参数名不一致：`/book/bookmarklist` 用大写 `bookId`，`/review/list/mine` 用全小写 `bookid`。**别写错**。

`/review/list/mine` 翻页：`hasMore=1` 时下一页带 `synckey`，循环到 `hasMore=0` 拼齐所有 `reviews[]`。

把两边数据按 `chapterUid` 用 `chapters[]`（来自 `/book/bookmarklist` 的同次返回）查 `title`，按章节分组。

#### 下钻展示格式（聊天界面渲染，不是写入 flomo 的内容）

```
📖 《<书名>》 — <作者>
📊 X 条划线，Y 条想法

### 第 N 章「<章节标题>」（n 条）

> <划线原文>

> <另一条划线原文>
> 想法：<针对这条划线的想法正文>

### 第 N+1 章「<下一章节标题>」（k 条）

> ...
```

注意：
- 这里使用 markdown `>` blockquote 是**给聊天 UI 的**，不是写入 flomo 的内容。flomo 不支持 `>`。
- `markText` 与 `review.content` 都做去首尾空白后展示，内部换行原样保留。
- `chapterUid` 在 `chapters[]` 找不到对应项时，章节标题写「未知章节」。
- 整本书评（`isFinish=1` 且没有 `chapterUid`）放在最后一个「全书」段。
- 长内容**不要**截断。逐字呈现。

### 2.3 下钻后的菜单

展示完笔记后，**必须**提示：

> 要把这些导出到 flomo 吗？
> - 回复「全部导出」→ 整本书的划线 + 想法都写入 flomo（流程见 §4）
> - 回复「只导划线」/「只导想法」→ 按类型筛选导出
> - 回复具体一条的引用片段 → 单条导出（流程见 §3）
> - 回复「换一本」→ 回到列表
> - 回复「先不导」→ 结束

「换一本」时，把刚才已拉的 `chapters[]`/`bookmarklist`/`reviews[]` 数据丢弃，重新提示用户从概览选；不要默认地循环。

## 3. 单条导出

适用：用户在浏览或别处见到某条具体划线/想法，明确要导这一条。

### 3.1 定位

如果上下文里已经能找到具体的 `bookmarkId`（划线）或 `reviewId`（想法），直接用。

否则让用户贴片段：

> 请贴一下你想导出的内容片段（10 个字以内即可），或者告诉我书名 + 章节，我帮你定位。

收到片段后：
1. 如果用户说了书名 → 先按 §4.1 拿到 `bookId`
2. 调 `/book/bookmarklist`（划线）或 `/review/list/mine`（想法），在返回的 `markText` / `review.content` 里做子串匹配
3. 多个候选时编号让用户选；零个时说「没找到，请确认书名或贴更具体的片段」

### 3.2 去重

按 `dedup.md` 的方式调一次 `memo_search`，命中即告诉用户「这条已经在 flomo 里了，跳过」并停止。

### 3.3 写入

按 `format.md` 渲染单条 memo 模板，调 `memo_create`，把生成的 memo 信息汇报给用户。

## 4. 单本导出

### 4.1 识别书

- 用户给 bookId → 直接用
- 用户给书名 → 调 `/store/search`：
  ```json
  {"api_name":"/store/search","keyword":"<书名>","scope":10,"skill_version":"<v>"}
  ```
  从 `results[0].books` 取前 3 个候选，编号给用户选；只有 1 个就直接用并告知。

### 4.2 拉笔记

并行调用：
- `/book/bookmarklist` — 取划线 `updated[]` 与 `chapters[]`
- `/review/list/mine` — 取想法 `reviews[]`，翻页直到 `hasMore=0`
- 书名：取 `/book/bookmarklist` 返回的 `book.title`；若缺，再调 `/book/info` 补一次

### 4.3 去重 + 清单

按 `dedup.md` 的批量优化方式：先 `memo_search keywords:"wr:hl" tag:"微信读书/{清洗后书名}" limit:200` 拿到这本书已存在 memo 的 ID 集合，本地比对每条划线/想法。

把汇总展示给用户：

```
准备导出《<书名>》：
- 划线 X 条 → 新增 a 条，跳过 b 条（已存在）
- 想法 Y 条 → 新增 c 条，跳过 d 条
- 标签：#微信读书 #微信读书/划线 或 #微信读书/想法 #微信读书/{清洗后书名}

是否继续？（回复「确认」开始写入）
```

收到「确认 / 导出 / OK」之前**绝不**写入。

### 4.4 写入

按 `format.md` 模板逐条 `memo_create`：

```
tool: memo_create
arguments:
  content: <按模板渲染的正文>
```

不传 `created_at`；让 flomo 用当前时间，方便用户在「每日回顾」里看到本次导出。

错误处理：
- 单条失败 → 等 2 秒指数退避重试一次。仍失败计入 `failed[]`，继续下一条
- 章节标题缺失 → 写「未知章节」占位，不跳过
- 连续 5 次 `memo_create` 全失败 → 停止整批，告知用户

### 4.5 汇报

```
✓ 《<书名>》导出完成
- 新增：N 条（划线 a / 想法 b）
- 跳过：M 条（已存在）
- 失败：K 条
查看：https://flomoapp.com/mine
```

`failed[]` 非空时列出 `bookmarkId/reviewId` + 错误摘要。

## 5. 全部导出

### 5.1 拉全部笔记本

按 §2.1 调 `/user/notebooks` 翻页，**循环到 `hasMore=0`**（不要只取一页）。

### 5.2 总清单

对每本书都按 `dedup.md` 的批量优化跑一次 `memo_search`（标签为 `微信读书/{清洗后书名}`），构造每本书的「待新增」数。

汇总展示：

```
准备全量导出（共 N 本有笔记的书）：

| # | 书名 | 待新增 | 已存在跳过 |
|---|------|--------|-----------|
| 1 | 《XX》 | 38 | 4 |
| 2 | 《YY》 | 12 | 0 |
| ... | ... | ... | ... |

合计：待新增 a 条 / 已存在 b 条 / 共预计调用 memo_create a 次。

是否继续？（回复「确认」开始；也可以回「跳过 #2 #5」「只导前 10 本」之类微调）
```

**必须**用户显式确认；从不静默跑全量。

### 5.3 逐本写入

按用户确认后的清单**逐本**进入 §4.4 的写入流程。每完成一本立刻按 §4.5 格式汇报一次（不要等全部跑完），让用户能中途打断。

每本之间不需要 sleep；如果连续 5 次 `memo_create` 全失败，停止整批，告诉用户。

### 5.4 全量汇报

全部跑完后，给最终汇总：

```
✓ 全量导出完成
- 涉及书籍：N 本
- 新增：a 条（划线 x / 想法 y）
- 跳过：b 条
- 失败：c 条
```

---

## 6. 通用规约

- weread 接口的所有参数**平铺**在 JSON body 顶层，与 `api_name`、`skill_version` 同级；不要包在 `params`/`data`/`body` 对象里
- 时间戳字段（`createTime`、`updateTime`、`finishTime`）展示时转 `YYYY-MM-DD`，不要直接显示原始数字
- 阅读进度 `readingProgress` 是 0–100 整数，展示时带 `%`
- 所有跨次会话的内容（如用户已选书名、bookId、概览）只保留在当前对话上下文中；不要写到 flomo memo 之外的任何持久化位置
